# Feature Specification: Agent Orchestrator — GitHub PR Review (v1)

**Feature Branch**: `dawid/advanced-orchestrations` (existing — speckit-git-* hooks skipped per project memory)
**Created**: 2026-05-10
**Status**: Draft
**Input**: User description: "Build an agent-orchestration daemon that runs teams of agents in response to triggers, with a live console dashboard and phone push notifications. v1 ships one vertical slice: a configured GitHub repo is polled for new pull requests, an automated review is generated by a peer agent and posted back as a PR comment, and the operator is notified at every meaningful step on screen and on their phone."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Automated PR Review on a Configured Repo (Priority: P1) 🎯 MVP

The operator configures the orchestrator to watch a single GitHub repository. When a contributor opens (or updates) a pull request on that repo, the orchestrator detects the event within the configured polling interval, dispatches the review job to a running peer Claude Code agent, and posts the agent's review back to the pull request as a comment via the `gh` CLI. The operator does nothing during the run — the pipeline is fully autonomous from "PR opened" to "review posted".

**Why this priority**: This is the headline capability of v1. Without it, no value is delivered. Every other story exists to make this autonomous run observable, recoverable, and trustworthy.

**Independent Test**: Configure the orchestrator against a throwaway test repo, start it, push a branch, and run `gh pr create`. Within the configured polling interval plus the agent's run time, `gh pr view <n> --comments` shows a review comment authored by the orchestrator's agent identity, and the review covers the diff content (not boilerplate).

**Acceptance Scenarios**:

1. **Given** a configured repo with the orchestrator daemon running and a peer Claude Code adapter reachable, **When** a pull request is opened on the repo, **Then** within one polling interval the daemon emits a `run.started` event referencing the PR number and head SHA, dispatches the review job, and ultimately posts a comment on the PR via `gh`.
2. **Given** a pull request is updated (new commits pushed) after a prior review run, **When** the next poll detects the new head SHA, **Then** the daemon starts a fresh run for the updated commit and posts an additional review comment without overwriting the prior one.
3. **Given** the peer agent is unreachable or returns an error, **When** the dispatch fails, **Then** the daemon emits a `run.failed` event with an actionable error and does NOT post a comment on the PR.
4. **Given** the orchestrator is configured with credentials for a single repo, **When** a PR is opened on a different repo, **Then** the daemon ignores it.

### User Story 2 — Live Console Dashboard (Priority: P2)

The operator launches a separate dashboard process and sees a live view of agent runs in progress: a header banner identifying the orchestrator, a table of recent runs with status and PR reference, and a tail of structured events as they arrive. The dashboard updates in near real time as the daemon emits events. The dashboard is read-only — it observes; it does not control.

**Why this priority**: Without a dashboard, the operator must tail logs or query the database manually to know what the orchestrator is doing. The dashboard converts an opaque background process into something the operator can trust at a glance. It is required for normal operation but not strictly required to deliver a single review (the daemon alone can post one).

**Independent Test**: With the daemon running and a PR-review run in progress, launch the dashboard. The runs table populates with the in-flight run, the run's status transitions through `running` to `completed` (or `failed`) on screen, and the event tail shows the agent's streamed output ordered by time.

**Acceptance Scenarios**:

1. **Given** the daemon has produced events, **When** the dashboard launches, **Then** the runs table is populated with historical runs (most recent first) and the event tail shows the most recent events.
2. **Given** the dashboard is open and the daemon is running, **When** the daemon emits a new event, **Then** the dashboard reflects it visibly within one second without operator interaction.
3. **Given** the dashboard is open, **When** the daemon process is killed, **Then** the dashboard remains usable (showing the historical record); when the daemon is restarted, new events appear in the tail.
4. **Given** the dashboard is closed and reopened, **When** it starts, **Then** it shows the same historical runs and events from before the close.

---

### User Story 3 — Phone Push Notifications at Each Step (Priority: P2)

At each meaningful state transition (run started, run completed, run failed, agent requested input), the orchestrator sends a push notification to the operator's phone via a public push service (ntfy.sh). The notification contains a level (info / warn / error / needs-input), a short title, a one-line body, and (when applicable) a link to the PR. The operator subscribes their phone to a topic name configured out-of-band; no account is required on the push service.

**Why this priority**: The operator wants to walk away from their machine and still know what happened. Phone push closes that loop. It is a P2, not a P1, because the dashboard already covers operator-at-the-desk visibility — phone push extends that surface, it does not replace it.

**Independent Test**: Configure the topic, subscribe a phone in the ntfy mobile app, run a PR-review end-to-end. The phone receives at least two notifications: one when the run starts and one when the run finishes (success or failure). Notification levels match the actual outcome.

**Acceptance Scenarios**:

1. **Given** a topic is configured and a phone is subscribed, **When** a PR-review run starts, **Then** the phone receives an info-level notification within 5 seconds containing the PR number and repo.
2. **Given** the run completes successfully, **When** the review comment is posted, **Then** the phone receives an info-level notification with a clickable link to the PR.
3. **Given** the run fails, **When** the daemon records the failure, **Then** the phone receives an error-level notification (high priority) with a one-line failure reason.
4. **Given** the push service is unreachable, **When** the daemon attempts to send a notification, **Then** the daemon logs the push failure as a non-fatal warning and continues; the dashboard still reflects the underlying event.

---

### User Story 4 — Replayable History Across Restarts (Priority: P3)

When the daemon, the dashboard, or the operator's machine is restarted, the historical record of past runs and events survives. The operator can inspect what the orchestrator did yesterday without re-deriving it from GitHub or logs.

**Why this priority**: Useful for trust and post-mortems but not required to ship a single autonomous review. v1 only needs durable history at the event level — in-flight tasks at the moment of crash are not resumed (they restart from scratch on the next poll).

**Independent Test**: Run several PR-reviews, kill the daemon, kill the dashboard, reboot. Relaunch both. The historical runs and event tail still render the prior activity.

**Acceptance Scenarios**:

1. **Given** prior runs exist in the historical record, **When** the dashboard relaunches after any kind of process restart, **Then** the historical record renders identically to what it showed before the restart.
2. **Given** the daemon is killed during an in-flight run, **When** it is restarted, **Then** the in-flight run does NOT resume; instead, the next poll re-detects the still-open PR and starts a fresh run.

---

### Edge Cases

- A pull request is closed (or merged) between the poll detecting it and the review being posted. The daemon attempts to post; if `gh` reports the PR closed/merged, the daemon emits a `run.skipped` event with the reason and does NOT mark the run as failed.
- The configured peer adapter is not running when the daemon starts. The daemon does not crash; it logs and notifies on the first dispatch attempt and continues polling.
- `gh` CLI authentication has expired. The daemon detects the auth failure (non-zero exit + characteristic stderr) on first use, emits an `auth.failed` event, sends an error-level notification, and pauses the polling loop until the operator restarts the daemon.
- The peer agent returns a review that exceeds GitHub's comment size limit. The daemon truncates with a clear marker and a link to the full review payload in the event log.
- Two PRs are opened in the same poll window. Each is dispatched as its own independent run with its own event stream — runs do not share state.
- The push service rejects the request (bad topic, rate-limited). Treated as non-fatal: log a warning, surface in dashboard, do not abort the run.
- The historical event store is unavailable or corrupted at daemon start. The daemon refuses to start and surfaces an actionable error rather than silently dropping events.
- The daemon process is killed during a dispatch. On restart, the prior run is left in `running` status in the historical record (no resume); the operator sees it as orphaned in the dashboard, and the next poll handles the still-open PR independently.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST allow configuration of exactly one GitHub repository to watch per running daemon, identified by `owner/repo`.
- **FR-002**: System MUST detect newly-opened and newly-updated pull requests on the configured repository within a configurable polling interval (default 30 seconds).
- **FR-003**: System MUST treat each new head commit on a pull request as a distinct review-eligible event, even if a prior review for an earlier commit on the same PR has already been posted.
- **FR-004**: System MUST dispatch each detected pull-request event to a peer Claude Code agent over the existing A2A delegation channel, supplying the PR number, head SHA, repo, and the diff text.
- **FR-005**: System MUST post the peer agent's final output as a pull-request comment via `gh pr review --comment -F -` (or equivalent `gh` invocation), authored by the credentials available to the local `gh` CLI.
- **FR-006**: System MUST persist every meaningful state transition (run started, agent message received, run completed, run failed, run skipped, auth failure, push failure) to a durable, append-only historical record that survives daemon, dashboard, and machine restarts.
- **FR-007**: System MUST expose the historical record to a separate dashboard process via a read-only, polling-friendly interface (no requirement that the dashboard share a process or address space with the daemon).
- **FR-008**: System MUST send a push notification to a configured topic on a public push service for every run-level state transition (started, completed, failed) and for any auth or dispatch failure, with notification level matching the severity of the event (info / warn / error / needs-input).
- **FR-009**: System MUST treat push-notification failures as non-fatal — the underlying event MUST still be recorded in the historical record and visible in the dashboard.
- **FR-010**: System MUST expose two operator-facing entry points: a daemon command that runs the trigger-agent-publisher pipeline, and a dashboard command that renders the live view.
- **FR-011**: System MUST read all secrets and configuration values (push topic, A2A peer URL, A2A bearer token, repo identifier, polling interval, historical-record path) from process environment only — the system MUST NOT create, modify, or read `.env` files.
- **FR-012**: System MUST NOT post a review comment on a pull request whose dispatch returned an error from the peer agent.
- **FR-013**: System MUST NOT resume in-flight runs after a daemon restart; the historical record MUST mark such runs as orphaned, and the next poll handles the still-open PR independently.
- **FR-014**: System MUST expose the trigger surface as a strict abstraction so future trigger types (webhook receiver for GitHub, GitHub Issues, manual dispatch) can be introduced without modifying the core dispatch pipeline.
- **FR-015**: System MUST be runnable on a single developer machine behind NAT — no public ingress, no inbound port from the internet, no third-party hosting required for v1.
- **FR-016**: System MUST NOT introduce a new global skill or agent under `global/` to satisfy v1; orchestration logic lives in its own service package and reuses existing skills/agents (`/git`, `/pr`, `/review`, `@code-plan-verifier`) by invoking them inside dispatched prompts.
- **FR-017**: Every deviation from the published A2A specification surface that the orchestrator depends on MUST be recorded as an ADR per Constitution Principle I — silent deviation is forbidden.

### Key Entities *(include if feature involves data)*

- **Trigger Event**: A normalized record describing something the orchestrator should react to. Has a kind (`pr.opened`, `pr.updated`), a repo identifier, a PR number, a head SHA, a payload, and a detection timestamp. Produced by a trigger source; consumed by the dispatch pipeline.
- **Run**: A single end-to-end execution of an agent in response to one Trigger Event. Has a unique id, a reference to the Trigger Event that started it, a lifecycle state (`pending`, `running`, `completed`, `failed`, `skipped`, `orphaned`), a start timestamp, an end timestamp (when terminal), and a reference to the produced artifact (the review comment).
- **Event**: A timestamped, structured record emitted during a Run. Has a kind (e.g., `run.started`, `agent.message`, `gh.review.posted`, `run.failed`), a level (`info` / `warn` / `error` / `needs_input`), a payload (free-form JSON), and the run id it belongs to.
- **Historical Record**: The durable append-only store of Events. Survives daemon, dashboard, and machine restarts. Read by the dashboard; written by the daemon.
- **Notification**: A push-service message derived from an Event whose kind warrants out-of-band attention. Has a level (mapped to push priority), a title, a one-line body, and an optional click-through URL (e.g., the PR URL).
- **Trigger Source**: A pluggable producer of Trigger Events (v1 ships exactly one: a polling source for GitHub pull requests). Future implementations (webhook receiver, GitHub Issues source) MUST plug in without touching the dispatch pipeline.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: From `gh pr create` on the watched repo to the operator's phone showing the "review starting" notification, no more than the configured polling interval plus 5 seconds elapse (default ≤ 35 s).
- **SC-002**: From "review starting" to a review comment being visible on the PR via `gh pr view <n> --comments`, no more than 90 seconds elapse for diffs under 500 lines.
- **SC-003**: The dashboard reflects new events emitted by the daemon within 1 second of the daemon writing them.
- **SC-004**: An end-to-end first-time setup (clone, env vars, two terminal commands, dashboard launch, push test PR) takes a new operator no more than 10 minutes when `gh` and the peer adapter are already configured.
- **SC-005**: The daemon survives 24 hours of continuous polling against an idle repo without crashing or leaking processes.
- **SC-006**: A simulated peer-agent failure produces an error-level phone notification, a `run.failed` event in the historical record, and zero review comments posted on the PR — in 100% of attempts across at least 5 trials.
- **SC-007**: A simulated daemon kill mid-run leaves the run marked as `orphaned` in the historical record (not `running` and not `completed`) within 60 seconds of the next dashboard refresh after restart, in 100% of attempts.
- **SC-008**: After daemon, dashboard, and machine restart, 100% of historical events written before the restart are still visible in the dashboard.

## Assumptions

- The operator has the GitHub CLI (`gh`) installed and authenticated for the watched repo with permission to post review comments.
- A peer Claude Code A2A adapter is already running locally (out of scope for this feature) and reachable via the configured A2A peer URL with a valid bearer token.
- The watched repo is one the operator owns or has been granted review-comment permission on; the orchestrator does not implement its own GitHub auth.
- A single daemon instance watches exactly one repo. Multi-repo, multi-tenant, and team-scale deployments are out of scope for v1.
- The historical record lives under the operator's home directory (within the existing Claude Code data area) on the operator's machine; multi-host replication is out of scope.
- The push service is the public ntfy.sh server using a topic name chosen by the operator (treated as a low-grade secret — anyone with the topic name can publish or subscribe). Self-hosting the push service is out of scope for v1.
- "Review" means a single PR comment authored by the agent; multi-comment reviews, line-level annotations, and required-changes blocks are out of scope for v1.
- The PR-fix agent (auto-applies review feedback), the Issue → Plan → PR agent, and any LangChain/LangGraph multi-agent state machine are explicitly out of scope for v1, but the trigger and dispatch surface MUST not preclude them.
- A webhook-based trigger is out of scope for v1 but the trigger abstraction MUST allow it as a drop-in replacement.
- The orchestrator service does NOT introduce a new global skill or agent; it reuses existing ones by invoking them inside dispatched prompts.
- The orchestrator service is a sibling of `services/a2a-bridge/` in the same monorepo, sharing the same toolchain and conventions.
- The operator runs on a Windows or POSIX workstation behind NAT with no public ingress.
