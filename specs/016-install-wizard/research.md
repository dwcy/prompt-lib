# Research: Installable Distribution with Installation Wizard

**Date**: 2026-07-11 | **Plan**: [plan.md](plan.md)

## R1 — PyPI distribution name

- **Decision**: The first release CANNOT ship as `cabal` on PyPI. Recommended distribution name: **`cabal-panel`**, keeping `cabal` as the import package and console command. Final call is the user's, recorded as a release-runbook step; `pyproject.toml` keeps `name = "cabal"` until the user confirms.
- **Rationale**: Verified live on 2026-07-11 via `https://pypi.org/pypi/cabal/json`: an unrelated package `cabal` exists (exact computable real numbers, v0.2.1, uploaded 2026-06-15) and is actively maintained — the name is not claimable, and PEP 541 name-transfer does not apply to active projects. PyPI distribution name, import package name, and console-script name are independent, so `cabal-panel` on PyPI can still install `import cabal` + the `cabal` command; this keeps every internal path, the exe name, and muscle memory intact. Note: the `cabal` *command* collides with Haskell's `cabal` binary on machines with a Haskell toolchain — acceptable for the primary audience (owner's machines, no Haskell), documented in the runbook as a known trade-off with `cabal-panel` as an optional second alias if it ever bites.
- **Alternatives considered**: Full rename to `prompto` (floated in `docs/release-readiness.md`) — larger blast radius (package dir, exe, docs, muscle memory) for no functional gain; can still be done later. Namespaced `dwcy-cabal` — ugly, non-descriptive. Contacting the `cabal` owner for the name — active project, no realistic path.

## R2 — Non-interactive mode design (FR-014)

- **Decision**: Add an argparse layer in `cabal/__main__.py`. Zero args → launch the Textual wizard exactly as today. Subcommands: `cabal apply [--components <keys>] [--dry-run] [--yes]`, `cabal doctor [--json]`, `cabal uninstall [--restore-backups] [--dry-run] [--yes]`, `cabal --version`. Headless runners live in a new `cabal/headless.py` that calls the same service modules the TUI uses (`diff_apply`, `config_doctor`, `uninstall_service`) — no duplicated logic. Destructive subcommands without `--yes` print the preview and exit with code 3 (confirmation required) instead of prompting, so they are safe in pipelines.
- **Rationale**: FR-014 requires answers supplied up front with results identical to the interactive path — the only way to guarantee "identical" is to share the service layer and make the CLI a thin invoker. argparse is stdlib (no new dependency, matters for the frozen exe). Exit-code contract makes provisioning scripts able to distinguish "up to date" / "changes applied" / "needs confirmation" / "failure".
- **Alternatives considered**: Typer/Click — new runtime dependency bloating wheel + exe for four subcommands; rejected. Environment-variable answers file — less discoverable than flags, harder to validate; rejected. Textual headless mode — runs the full TUI event loop for no benefit; rejected.

## R3 — Installation manifest format and location (FR-008, FR-015, FR-017)

- **Decision**: JSON manifest at `~/.claude/.cabal/install-manifest.json`, written by every apply (wizard or headless). Schema (detailed in `data-model.md`): tool version, source mode (source/wheel/frozen), timestamp, selected component keys, per-file records (component key, relative path, SHA-256, backup ref if one was taken), and a `status` field journaled `in_progress` → `complete` around the write phase.
- **Rationale**: A dot-directory inside the deploy target keeps the record adjacent to what it describes (survives repo moves, works for wheel/exe installs with no repo), while `.cabal/` is invisible to Claude Code's config loading. SHA-256 per file lets doctor distinguish "modified by hand" from "stale version" without keeping full copies. The status journal is the cheapest interrupted-apply detector that satisfies FR-017: any later run seeing `in_progress` knows the previous apply died mid-write.
- **Alternatives considered**: SQLite — overkill for a few hundred rows, adds binary state to `~/.claude/`; rejected. Manifest inside the package install dir — lost on `uv tool uninstall`, and there can be N installs but only one deploy target; rejected. Reusing `cleanup_service`'s backup `manifest.json` — different concern (extras cleanup), merging them would tangle two lifecycles; rejected.

## R4 — Interrupted-apply detection and recovery (FR-017)

- **Decision**: Journal-based: write the manifest with `status: "in_progress"` (including the planned file list and backup dir) before the first file write, flip to `complete` as the last step. On any subsequent apply/doctor/uninstall run, an `in_progress` manifest triggers a recovery offer: **resume** (re-run the apply — the operation is idempotent since sources are compared by content) or **roll back** (restore the recorded backups). The wizard surfaces this as a modal; headless mode reports it and exits with a distinct code unless `--yes` resumes.
- **Rationale**: The apply operation is already idempotent (diff-driven, content-compared), so "resume" is simply "apply again" — no partial-write bookkeeping needed beyond knowing an apply was interrupted. Rollback reuses the existing timestamped-backup machinery.
- **Alternatives considered**: Atomic staged deploy (write to temp tree, swap) — `~/.claude/` is a live directory other tools hold open on Windows; directory swaps are not atomic cross-platform and break file locks; rejected. No detection (rely on idempotency alone) — violates FR-017's "never report a partial deployment as successful"; rejected.

## R5 — Uninstall scope (FR-013)

- **Decision**: New `uninstall_service.py`, manifest-driven: remove exactly the files listed in the latest `complete` manifest (verifying they are still cabal-managed — if a file's hash matches neither the manifest hash nor current source, treat as user-modified and ask/skip), offer restoration of recorded pre-install backups, then remove `~/.claude/.cabal/` itself last. When no manifest exists (installs predating this feature), fall back to the component diff (`Component.list_files()` ∩ disk) with an explicit "legacy uninstall — review the list" warning. The uninstaller never touches files outside `COMPONENTS` destinations and never recursively sweeps directories (same rule `cleanup_service` already follows).
- **Rationale**: Manifest-first satisfies FR-010/FR-013 precisely (only managed files); the legacy fallback keeps the feature useful on machines deployed before the manifest existed. Removing the manifest dir last means a crash mid-uninstall is detectable and resumable.
- **Alternatives considered**: `uv tool uninstall cabal` only — removes the tool but leaves `~/.claude/` deployed; that is package removal, not config uninstall, and both are documented as distinct steps in quickstart. Recursive delete of component dirs — would destroy user-authored agents/skills living beside managed ones; forbidden.

## R6 — Release channel and publishing mechanics

- **Decision**: Keep the existing three-channel release (`.github/workflows/release.yml`): PyPI wheel+sdist via Trusted Publishing (OIDC, `pypi` environment), Windows one-file exe, GitHub Release aggregating all artifacts. The feature adds `docs/release-runbook.md` covering the user-owned steps in order: (1) confirm distribution name (R1), (2) update `pyproject.toml` + `release.yml` URL/globs if renamed, (3) create the PyPI project + Trusted Publishing config (publisher: this repo, workflow `release.yml`, environment `pypi`), (4) create the `pypi` GitHub environment, (5) run `python scripts/test-all.py --strict-missing`, (6) `python -m build` + wheel content inspection (the four required paths from `docs/release-readiness.md`), (7) exe smoke test on Windows, (8) tag `v0.1.0`. No tag is created by this feature.
- **Rationale**: The workflow already implements the right mechanics; what was missing is the ordered, user-executable procedure and the name blocker being surfaced. Trusted Publishing avoids long-lived PyPI tokens in repo secrets. Tagging stays manual per house rule (never push without being asked) and per `docs/release-readiness.md`'s tagging rule.
- **Alternatives considered**: GitHub Releases as the only channel (skip PyPI) — loses `uv tool install <name>` one-command UX that FR-001 targets; kept as an interim channel since wheels attach to the Release regardless. Publishing under an API token — inferior to OIDC Trusted Publishing; rejected.

## R7 — Wizard gap check against spec user stories

- **Decision**: The existing wizard already covers: prerequisite-free launch (deps bundled in wheel/exe), component multi-select, dry-run diff preview, timestamped backups, restore screen, env-var init (writes machine env, not env files), config doctor, drift markers. Gaps to implement are limited to: manifest recording (R3), uninstall screen (R5), interrupted-apply modal (R4), and post-install verification summary (a final "N files deployed, M backed up, doctor clean" report satisfying US1 acceptance 1). No re-architecture of existing screens.
- **Rationale**: Direct code inspection of `setup/src/cabal/` (diff_apply.py, cleanup_service.py, config_doctor.py, components.py, README) shows the P1/P2 stories are largely built; building parallel "installer" screens would violate the house rule against duplicating existing surfaces.
- **Alternatives considered**: A separate `cabal install` first-run wizard distinct from the update flow — the flows are identical (fresh machine is just "everything is NEW in the diff"); rejected as duplication.
