Executive summary

Four high-value ideas, one strategic caution

OpenHands is now an ecosystem rather than one clean monorepo. Its most relevant work for prompt-lib is in standards-compatible configuration, real file-based delegation, portable hooks, and observable agent runs.

01
Best direct fit

Generate OpenHands-native agents and skills from prompt-lib’s existing catalog.

02
Best architecture idea

Separate reusable policy from harness-specific hook payload adapters.

03
Largest current gap

Add behavioral evaluations and outcome traces, not only structural validation.

04
Useful later

Use ACP/Agent Canvas as an optional runner, never as prompt-lib’s core.

Bottom line: prompt-lib already has the richer curated knowledge library. OpenHands has the more portable execution contract. The opportunity is to connect them, not replace one with the other.
What OpenHands is doing

A control plane, an SDK, and an extension ecosystem

The repository named by the request is in an active transition. Its README now describes Agent Canvas and explicitly points the agent/server engine and Canvas UI to separate repositories.

Control plane

Agent Canvas

Starts conversations, switches backends, manages settings, and creates scheduled or webhook-driven automations.

Agent loop, tools/MCP, skills, plugins, subagents, security policy, context condensation, and persistence.

Workspace options

Local or isolated

The same conversation API can target the local filesystem or a remote Docker/Kubernetes workspace.

External agents

ACP adapters

Canvas can drive Claude Code, Codex, Gemini CLI, or a custom stdio ACP server while each external agent keeps its own tools and configuration.

Automation

Schedules and events

An automation server can start agent work from cron schedules, GitHub events, Slack, Linear, and other integrations.

Distribution

Extensions registry

A public catalog packages skills and executable plugins, and demonstrates parallel OpenHands, Claude, and Codex manifests.

Important transition risk: do not design against old paths in OpenHands/OpenHands. The forward-looking implementation lives mainly in software-agent-sdk, agent-canvas, automation, and extensions. Capability detection and pinned compatibility tests matter more than copying today’s folder tree.

Ranked recommendations

What should be brought into prompt-lib

These are ordered by practical value for this repository, not by how prominent the feature is in OpenHands.

Add an OpenHands export target with real subagents

Generate OpenHands-native .agents/skills, .agents/agents, and plugin metadata from canonical prompt-lib assets. Map Claude tools such as Read/Write/Edit/Glob/Bash to OpenHands file_editor and terminal; add model: inherit; preserve each agent body and routing description.

  • Build on the existing Codex conversion manifest instead of hand-maintaining a third duplicate tree.
  • Emit a target capability report for unsupported fields such as Claude-only subagent events.
  • Add schema and smoke tests using a pinned OpenHands SDK version.
High impactMedium effort

Split hook policy cores from harness adapters

OpenHands and Claude share a large hook lifecycle subset, but their tool names and input contracts differ. Keep pure functions such as command_guard.scan() as the policy core, then add thin Claude/OpenHands adapters that normalize events into a small internal shape.

  • Start with command safety, branch protection, file-write guard, and stop quality gates.
  • Keep deterministic policy authoritative; treat LLM-provided risk labels only as an additional signal.
  • Test allow/block parity with fixtures for both payload formats.
High impactMedium effort

Create a harness-neutral behavioral eval suite

Structural tests prove that assets deploy; they do not prove that an agent routes correctly, activates the right skill, blocks a dangerous action, or satisfies a stop condition. Borrow OpenHands’ benchmark discipline without importing its large benchmark stack.

  • Define small cases with prompt, available assets, expected routing/tool policy, and pass criteria.
  • Begin with six critical flows: agent routing, skill activation, destructive command block, branch guard, stop gate, and conversion drift.
  • Keep deterministic cases in normal CI; make model-backed quality cases opt-in and record model/version.
High impactMedium effort

Make one canonical cross-harness capability matrix

OpenHands’ extensions repository publishes parallel OpenHands, Claude, and Codex manifests. prompt-lib should similarly describe each asset once, then generate target-specific manifests and record semantic loss.

  • Add target support fields to the conversion/OKF model: native, adapted, prompt-only, unsupported.
  • Generate .claude-plugin, .plugin, and future manifest views from the same metadata.
  • Surface drift in Cabal’s existing conversion and doctor panels.
High leverageSmall–medium effort

Feed PR review learnings into self-improvement

The OpenHands learn-from-code-review skill has a useful input source that prompt-lib’s correction-driven memory does not: recurring human review comments across merged pull requests. Adapt the workflow, not the file paths.

  • Require multiple similar comments before proposing a rule.
  • Cite PR/comment evidence and separate team convention from one reviewer’s preference.
  • Produce a reviewable proposal; never auto-edit always-loaded instructions from mined feedback.
Medium impactSmall effort

Extend the OKF usage ledger with redacted execution outcomes

OpenHands’ event stream shows how independent services can observe runs without mutating the agent loop. Add an opt-in, harness-neutral ledger that links OKF concept IDs to activation, outcome, block reason, and duration—without storing prompt bodies, file contents, secrets, or command output.

  • Use it to find unused skills, misrouted agents, noisy hooks, and repeated stop failures.
  • Keep raw harness logs outside OKF; ingest only redacted aggregates.
  • Defer until the behavioral eval schema is stable so both share outcome vocabulary.
StrategicLarge effort

Adoption boundaries

What should stay outside prompt-lib

OpenHands solves product/runtime problems that this repository deliberately does not own.

Do not copy

Agent Server, Canvas, and sandbox runtime

  • They introduce a long-running service, frontend, persistence, container security, and remote deployment burden.
  • Use OpenHands itself if that operational layer is wanted.
  • prompt-lib should only generate compatible configuration and optional integration metadata.
Do not trust alone

LLM-declared action risk

  • OpenHands can ask the model to include a risk field in tool arguments.
  • That is helpful for confirmation UX but cannot replace deterministic command/file guards.
  • Unknown actions should remain ask-gated or blocked, not silently downgraded.
Optional runner

ACP and Agent Canvas

  • ACP is attractive as a common control protocol for Claude Code, Codex, and Gemini.
  • It does not automatically inject prompt-lib content into those external agents; each harness still needs its own deployment.
  • Test it after OpenCode setup settles, as an operational companion rather than a conversion replacement.
Packaging caveat

Agent Canvas installation

  • The current official local quickstart uses a package-manager path that conflicts with prompt-lib’s pnpm/bun-only rule.
  • Do not add a Cabal installer until an approved, tested packaging route is chosen.
  • Docker remains operationally heavier and should be user-opted, never a default dependency.
Suggested implementation sequence

A clean path that does not disrupt active work

Finish the active OpenCode feature first. Then treat OpenHands support as a separate Spec Kit feature with explicit conversion and compatibility contracts.

Phase 0 · now

Record the decision only

Keep this report as research input. Do not modify feature 017 or mix OpenHands work into its OpenCode acceptance scope.

Phase 1 · spec

Specify a cross-harness asset model

Define native/adapted/unsupported states, tool mappings, manifest generation, hook-normalized events, and reversible install behavior.

Phase 2 · slice

Ship agents and skills first

Generate a minimal OpenHands plugin plus .agents/agents output, validate 2–3 representative agents, and expose preview/apply through Cabal.

Phase 3 · safety

Add hook adapters and parity tests

Port only the common lifecycle subset and verify the same commands/files receive the same allow/block result across harnesses.

Phase 4 · quality

Add behavioral evals and redacted outcomes

Use the eval suite to gate conversions; only then consider event-derived usage analytics or an ACP/Canvas experiment.

Recommended scope statement: “Add OpenHands as a generated consumer of prompt-lib assets; do not embed OpenHands runtime services.” That boundary keeps the feature small enough to finish and test.
