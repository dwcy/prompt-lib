Borrow the operating discipline.
Keep your architecture.

Verdict: loop-engineering has several excellent ideas for governing recurring agent automation—trust levels, durable state, budgets, attempt caps, kill switches, and a pattern registry. prompt-lib already has the stronger engineering substrate: enforced hooks, specialist routing, Spec Kit, worktree isolation, durable SQLite event history, multi-harness deployment, and OKF retrieval. The right move is to absorb the control model into Cabal and OKF, not adopt their npm toolchain or markdown layout wholesale.

03 · What they do that you do not

Five ideas worth bringing into prompt-lib

Prioritized by leverage and fit with the existing Cabal, OKF, Spec Kit, and orchestrator architecture.

A common automation contract

Define every recurring agent workflow with the same fields: trigger, cadence, watched scope, state store, trust level, verifier, permissions, gates, attempt cap, budget, kill switch, owner, and success metrics. This is the conceptual value of LOOP.md, without forcing one more root markdown file into every project.

P0 · Foundation

Mechanical budget and circuit-breaker policy

Add per-run and per-day limits plus repeated-error and no-progress detection. prompt-lib already records real events; enrich those events with attempt count, model/token accounting, stop reason, and degraded mode. A policy should stop or downgrade the run—not merely advise the model.

P0 · Safety

A shared trust ladder

Use L0/L1/L2/L3—or clearer local names such as Draft, Observe, Assist, Act—as a release gate for automation. Require a measured report-only period before enabling writes, and require a verifier, permission scope, budget, and kill switch before unattended operation.

P1 · Governance

An automation pattern registry and picker

Make Cabal answer “what should I automate next?” from machine-readable metadata. Reuse the existing catalog and OKF machinery: patterns become concepts with compatibility, risk, cost, prerequisites, starter, and evidence of live runs.

P1 · UX

A curated automation incident catalog

Promote operational lessons into a public, linked catalog: infinite retries, verifier theater, state rot, notification fatigue, scope overreach, comprehension debt, and cross-loop collisions. Your ADRs and self-improvement memory contain the raw material; they need a discoverable operating view.

P2 · Learning
04 · What you already do better

Do not regress these advantages

Any adoption should fit these strengths rather than introducing a parallel ecosystem.

Enforcement over aspiration

  • Unsafe shell shapes are blocked before execution.
  • Security guard files protect themselves from edits.
  • Background writers without worktree isolation are blocked.
  • Branch/session collision handling is automatic.

Durable, queryable operations

  • SQLite WAL event history is richer than a markdown run log.
  • Run state is derived from append-only events.
  • Orphan recovery and cursor state survive restarts.
  • A dashboard and phone notifications expose live behavior.

Specification discipline

  • Spec → plan → tasks → implementation → verification is explicit.
  • Contracts, data models, research, and measurable outcomes are versioned.
  • Parallel task ownership and isolation are encoded in templates.
  • Independent verifier and security audit stages are built into workflows.

Platform integration

  • Cabal deploys, diagnoses, previews, restores, and manages local config.
  • Claude, Codex, OpenCode, MCP, and A2A are integrated as real targets.
  • OKF adds graph-backed search, preflight, context packs, analytics, and usage.
  • Package security, recovery, setup drift, and conversion have tests and UI.
05 · Caveats

Good ideas, uneven proof

The repo is useful, but its marketing language and 100/100 dogfood score should not be treated as independent evidence of production readiness.

The readiness audit is gameable

loop-audit mostly checks file presence, known skill names, and regex hints such as “worktree,” “budget,” “human review,” or “max attempts.” Its “activity” proof can be satisfied by a timestamp, workflow filename, run-log file, matching commit text, or cadence wording. It does not validate that controls are wired into execution.

The flagship dogfood loop is not an agent loop

The scheduled daily-triage workflow runs deterministic Node and shell steps. It audits files, checks two workflow statuses, rewrites state, and opens a PR. It makes no LLM call, yet appends a fixed tokens_estimate: 52000. That makes its cost telemetry illustrative, not observed.

Policy and workflow conflict

loop-constraints.md says never auto-merge to main without human approval. The daily-triage workflow unconditionally enables gh pr merge --auto --squash after its inline gates. Even if branch protection later requires approval, the audit cannot detect this semantic conflict—and still reports 100.

Breadth is partly templated breadth

The ten tool examples and sixteen starters are useful onboarding material, but many are variants of the same markdown/skill payload. The implemented, repository-owned automations are narrower than the headline “seven production patterns” suggests.

Implication: adopt the concepts, schemas, and operator questions. Do not import the score, badge, hard-coded cost assumptions, or filesystem-presence logic as your source of truth.
06 · Suggested roadmap

Integrate the ideas without creating a second platform

A sequence that reuses Cabal, OKF, the orchestrator event log, existing worktree machinery, and Spec Kit.

Phase 1 · Contract

Define an AutomationDefinition schema

Fields: id, purpose, non-goals, trigger, cadence, scope, trust level, state store, verifier, tool scope, human gates, max attempts, run/day budget, kill switch, owner, and metrics. Store canonical definitions in one registry; export them to OKF as a new concept kind.

Deliverable: schema + doctor validation + one migrated PR-review definition.
Phase 2 · Guardrails

Add real budget and no-progress enforcement

Extend the orchestrator event model with attempts, token/cost observations, error fingerprints, breaker decisions, and degraded mode. Stop after repeated identical errors or an explicit cap. Prefer observed transcript/API usage; label estimates as estimates.

Deliverable: deterministic policy engine with tests and operator-visible stop reasons.
Phase 3 · Trust

Gate write authority by maturity level

Start every new pattern in Observe/report-only. Promote only after an evidence window: acceptable false-positive rate, bounded escalations, clean state pruning, and demonstrated kill-switch operation. Cabal should show why a pattern is or is not promotable.

Deliverable: readiness view based on validated configuration and actual run evidence.
Phase 4 · Catalog

Ship a small prompt-lib-native pattern set

Begin with existing strengths: PR review, issue triage, configuration-drift report, and release-note draft. Avoid CI auto-fix until breaker and budget controls are proven. Generate Claude/Codex/OpenCode bindings from one definition rather than copying starters.

Deliverable: Cabal pattern picker backed by the registry and OKF.
Phase 5 · Learning

Turn run evidence into operations knowledge

Dashboard actual runs, actionable findings, false positives, escalations, breaker trips, time-to-human-awareness, cost, and worktree outcomes. Link incidents to ADRs, self-improvement memory, and pattern guidance.

Deliverable: evidence-backed incident catalog and pattern health dashboard.
